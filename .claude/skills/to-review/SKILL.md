name: to-review
description: Create a pull request with all changes, move the issue to In Review status, and prepare for code review

You are running the `/to_review` command for the Chamuco App project. Your job is to create a pull request with all changes and move the associated issue to "In Review" status. This command should be used when work on an issue is complete and ready for code review.

## Prerequisites

Must be on an issue branch created with `gh issue develop` (branch name format: `<issue_number>-<slug>`).

## Step 1 — Run preparation script

Execute the automated preparation script:

```bash
./scripts/to-review.sh
```

The script will:
1. Validate the current branch is an issue branch
2. Check for uncommitted changes
3. Push the branch to remote
4. Generate diff context and save to `/tmp/to-review-<issue_number>-context.txt`
5. Read issue details and save to `/tmp/to-review-<issue_number>-issue.json`
6. Export `ISSUE_NUMBER` and `BRANCH_NAME` variables

**Handle exit codes:**
- Exit code 0: Success, proceed to PR creation
- Exit code 2: Uncommitted changes detected
  - Ask the user if they want to commit the changes first
  - If yes, use the standard commit process from CLAUDE.md (git commit instructions)
  - If no, abort with: ⚠️ There are uncommitted changes. Please commit or stash them before creating a PR.
- Any other exit code: Report the error and abort

If exit code is 2 and the user chooses to commit, re-run `./scripts/to-review.sh` after committing.

## Step 2 — Extract variables from script output

Parse the script's output to get:
- `ISSUE_NUMBER` (from the last line containing `ISSUE_NUMBER=`)
- `BRANCH_NAME` (from the last line containing `BRANCH_NAME=`)

## Step 3 — Generate PR title and body

1. Read the context files generated by the script:
   ```bash
   cat /tmp/to-review-${ISSUE_NUMBER}-context.txt
   cat /tmp/to-review-${ISSUE_NUMBER}-issue.json
   ```

2. Review the current conversation context to understand the "why" behind the changes.

3. Generate the title following these rules:
   - Maximum 72 characters
   - Conventional Commits prefix with optional scope: `feat(trips):`, `fix(auth):`, `docs:`, `refactor:`, `chore:`, `ci:`, `test:`
   - Imperative mood: "add", "fix", "update", "remove"
   - No period at the end
   - Capitalize only the first word after the colon

4. Generate the body with these sections (omit empty sections):
   - **Summary** — One paragraph explaining the motivation (the "why")
   - **Changes** — Grouped by logical intent, not by file
   - **Breaking Changes** — Only if applicable
   - **Testing** — Adapted to the type of change, always include edge cases
   - **Notes** — Only if there's something the reviewer must know

5. **IMPORTANT:** Add `Closes #<issue_number>` at the end of the body to automatically link and close the issue when the PR is merged.

## Step 4 — Create the pull request

Use a HEREDOC for the body to ensure proper formatting:

```bash
gh pr create --title "<generated_title>" --body "$(cat <<'EOF'
<generated_body>
EOF
)"
```

Save the PR URL from the output.

## Step 5 — Update issue status to "In Review"

Use the status update script:

```bash
./scripts/update-project-status.sh "${ISSUE_NUMBER}" "In Review"
```

If the script fails (non-zero exit code), note that the status can be updated manually in the GitHub UI. Do not block on this step.

## Step 6 — Output confirmation

Provide a clear confirmation message:

- ✅ Branch pushed to remote
- ✅ Pull request created: `<PR_URL>`
- ✅ Issue #`<issue_number>` status updated to "In Review" (or ⚠️ note to update manually if it failed)
- 🎯 The PR is now ready for code review

Keep the tone concise and actionable.
