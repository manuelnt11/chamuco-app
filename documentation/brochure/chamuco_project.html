1. Visión General del Proyecto

Propuesta de valor, audiencia y principios

1.1 La propuesta central

Chamuco no es una app de viajes. Es una app de grupos que viajan.

La distinción es fundamental. El sujeto de la plataforma no es el viaje — es el grupo de personas que lo hace junto. Los viajes van y vienen, pero los grupos acumulan historia, construyen cultura, desarrollan rankings internos y crean una identidad compartida que crece con los años. La referencia más cercana es Strava: una comunidad de personas que hacen algo juntas, lo rastrean, celebran hitos, ven el progreso mutuo y sienten el tirón del grupo como motivador. Chamuco hace eso para el viaje en grupo.

1.2 Problema que resuelve

Coordinar un viaje en grupo es inherentemente caótico. La información está dispersa en hilos de WhatsApp, hojas de cálculo y cadenas de correos. Chamuco App centraliza:

  • Quién va — gestión de participantes, invitaciones y confirmación de asistencia.
  • Cómo llegamos — reservas de transporte y seguimiento de movimientos.
  • Dónde nos quedamos — reservas de alojamiento y gestión de estadías.
  • Qué hacemos — planificación de actividades y construcción del itinerario.
  • Qué gastamos — seguimiento de gastos compartidos con liquidación al final.
  • Cómo nos comunicamos — chats grupales, canales y comunidad.
  • Qué hemos construido juntos — historia compartida, logros, reputación y el registro de cada viaje.

1.3 Audiencia objetivo

  • Grupos de viaje recurrentes: círculos de amigos, familias, clubes de viaje que organizan múltiples viajes al año.
  • Viajeros individuales que quieren construir una identidad pública de viajero y conectar con otros.
  • Organizadores que gestionan viajes en nombre de grupos y quieren que sus aportes sean reconocidos.

1.4 Principios clave

  • El grupo es el héroe — funcionalidades diseñadas alrededor de la relación a largo plazo entre personas.
  • Simplicidad primero — la UI debe ser intuitiva; planear debe sentirse más liviano que el caos actual.
  • Flexibilidad — un viaje puede ser tan simple como una salida de fin de semana o tan complejo como un itinerario multipaís.
  • Confianza y transparencia — gastos y estados de reservas visibles para todos los participantes relevantes.
  • Localización por diseño — multiidioma y multimoneda desde el primer día.
  • Validación en el mundo real — la gamificación recompensa el viaje real, no el engagement con la app.

Decisiones arquitectónicas críticas

PostgreSQL es la fuente de verdad para membresías. Firestore se usa exclusivamente para entrega de mensajes en tiempo real. Cualquier cambio de membresía debe sincronizarse inmediata y sincrónicamente a Firestore por el backend NestJS vía Firebase Admin SDK.
Firebase Admin SDK es el único escritor de Firestore. El frontend nunca escribe directamente. Todas las escrituras pasan por POST /api/v1/... en NestJS.
Sin sistema JWT personalizado. La autenticación se delega completamente a Firebase Authentication. El backend verifica tokens de Firebase ID vía admin.auth().verifyIdToken().
Orden del pipeline CI/CD: lint → type check → tests → build → imagen Docker → push → ejecutar migraciones DB → desplegar en Cloud Run.

  • Base path: /api/v1/
  • Autenticación: Bearer token (Firebase ID token) en header Authorization
  • Formato de respuesta: JSON. Errores siguen RFC 7807 (Problem Details).
  • Paginación: cursor-based en todos los endpoints de lista.
  • Swagger UI disponible en /api/docs (producción: protegido por IP o autenticación).

4.3 Reglas de documentación OpenAPI (obligatorias)

Todo endpoint de controlador NestJS debe tener:

  • @ApiTags, @ApiOperation (con summary), @ApiResponse para cada código HTTP posible.
  • @ApiBearerAuth() en endpoints autenticados.
  • @ApiProperty en todos los campos de DTOs (tipo, descripción, ejemplo, requerido/opcional).

6. Autenticación y Autorización

Firebase Authentication · Sin JWT personalizado

6.1 Firebase Authentication

La autenticación está completamente delegada a Firebase Authentication. El backend verifica Firebase ID tokens vía admin.auth().verifyIdToken(). No hay sistema JWT propio de Chamuco.

6.2 Métodos de login soportados (al lanzamiento)

  • Google Sign-In — OAuth 2.0 / OpenID Connect vía Firebase.
  • Facebook Sign-In — OAuth vía Firebase. Requiere una Facebook App registrada; el App ID y App Secret se almacenan en GCP Secret Manager.

Los usuarios pueden vincular ambos proveedores a la misma cuenta mediante el account linking de Firebase.

6.3 Flujo de verificación

  1. El cliente completa el flujo OAuth con Google o Facebook vía Firebase SDK.
  2. Firebase emite un Firebase ID token al cliente (idéntico independientemente del proveedor).
  3. El cliente envía el token en el header Authorization: Bearer <token>.
  4. El guard global de NestJS llama a admin.auth().verifyIdToken().
  5. Si el token es válido, el UID de Firebase se usa para buscar el usuario en PostgreSQL.
  6. En el primer login, se crea el registro del usuario con auth_provider = GOOGLE | FACEBOOK y el display_name se pre-llena desde el nombre de la cuenta del proveedor.
  7. El objeto usuario se inyecta en el contexto de la request (req.user).

6.4 Roles de plataforma

RolDescripción
USERRol por defecto. Acceso normal a la plataforma.
SUPPORT_ADMINCuenta de servicio para soporte técnico. Bypasa todas las restricciones de acceso a viajes y grupos (autenticación siempre requerida). Nunca se cuenta como participante. Cada escritura que realiza queda registrada inmutablemente en support_admin_audit_log. No asignable desde dentro de la app.

9. Usuarios y Perfiles Personales

Registro · Username · Perfil personal · Perfil de salud

9.1 Registro de usuario

Al crear una cuenta vía Firebase Authentication, el backend provisiona automáticamente: un registro en users, un registro en user_preferences (con defaults), un registro en user_profiles y un registro en user_gamification_stats.

9.2 Reglas del username

  • Debe ser único en la plataforma.
  • 3–30 caracteres. Solo a-z, 0-9, _, -. Sin mayúsculas.
  • Almacenado sin @, mostrado con @.
  • Obligatorio elegirlo en el registro — no se asigna automáticamente.

11. Participantes e Invitaciones

Caminos de entrada · Roles · Permisos · Lista de espera

11.1 Dos caminos de entrada

CaminoIniciado porRequiere visibilidad
Solicitud de uniónUsuarioSí — el usuario debe poder ver el viaje
InvitaciónOrganizadorNo — siempre disponible para cualquier usuario

Solo puede haber una solicitud activa o una invitación activa por usuario por viaje al mismo tiempo. Ambas pueden crearse y cancelarse a voluntad por quien las creó (el usuario retira su solicitud; el organizador revoca su invitación). Al cancelar, el registro se elimina.

11.2 Estados del participante

EstadoDescripción
PENDING_REQUESTSolicitud enviada por el usuario, esperando aprobación del organizador.
INVITEDInvitación enviada por el organizador, esperando respuesta del usuario.
CONFIRMEDParticipación confirmada.
REJECTEDSolicitud rechazada por el organizador.
DECLINEDInvitación declinada por el usuario.

11.3 Roles de viaje

RolDescripción
ORGANIZERControl total. Puede haber múltiples organizadores simultáneamente.
CO_ORGANIZERPermisos granulares definidos al asignar el rol.
PARTICIPANTParticipante estándar del viaje.

Rol vs. participación son independientes: un organizador puede o no viajar (is_traveling_participant). Los no-viajeros se excluyen de capacidad, división de gastos y presupuesto per-cápita.

11.4 Permisos del co-organizador

Al asignar CO_ORGANIZER, el organizador define explícitamente un conjunto CoOrganizerPermission[]:

  • MANAGE_ITINERARY · MANAGE_EXPENSES · MANAGE_PARTICIPANTS
  • MANAGE_RESERVATIONS · EDIT_TRIP_DETAILS · MANAGE_PRE_TRIP_TASKS
  • MODERATE_CHANNEL · VIEW_TRAVEL_PROFILES · AWARD_RECOGNITIONS

11.5 Invitaciones de rol

Cuando un organizador quiere agregar otro organizador o co-organizador, debe hacerlo mediante una invitación de rol (trip_role_invitations). Si el invitado ya es participante confirmado, la invitación solo actualiza su rol. Si no es participante aún, el organizador declara will_travel. Una invitación de rol con will_travel = true bypasa el modo lista de espera.

11.6 Modo lista de espera

La lista de espera no es una entidad separada — es una vista de registros PENDING_REQUEST ordenados por initiated_at. Cuando WAITLIST_MODE está activo y la capacidad está llena, el organizador debe aceptar solicitudes en orden cronológico (puede rechazar cualquier solicitud en cualquier momento, pero no puede saltarse una para aceptar una posterior).

13. Módulo de Gastos

Libro contable colaborativo · Sin dinero real · Estilo Splitwise
Chamuco no procesa dinero real. El módulo de gastos es un libro contable colaborativo. La liquidación se calcula al cierre del viaje y los participantes la ejecutan fuera de la app.

13.1 Dos capas

CapaTablaPropósito
Costos planificadostrip_budget_itemsDefinidos antes/durante el viaje. Alimentan el estimado de presupuesto.
Gastos realesexpensesRegistrados cuando se gasta dinero. El libro en vivo.

Cuando un ítem planificado se paga, se vincula al gasto real correspondiente.

13.2 Propiedad del gasto

ValorDivisiónGenera deuda
SHAREDDividido entre un conjunto de participantesSí
INDIVIDUALUna persona pagó por sí mismaNo (referencia en el libro)

13.3 Tipos de división

TipoDescripción
EQUALMonto total dividido por igual entre todos los incluidos.
EXACTCada participante tiene un monto exacto asignado.
PERCENTAGECada participante tiene un porcentaje asignado (suma = 100%).
SHARESDivisión por unidades proporcionales (ej. 1, 2, 3 partes).

13.4 Reglas de liquidación

  • Solo participantes con did_travel = true se incluyen en divisiones y liquidación.
  • Múltiples pagadores por gasto soportados (expense_payers). La suma de montos de pagadores debe igualar el total.
  • El campo exchange_rate_snapshot es inmutable — capturado al momento del registro.
  • Al completar el viaje, se calcula un grafo de deuda mínima (estilo Splitwise).

Los grupos son la unidad de identidad social de largo plazo. Un grupo puede tener múltiples viajes a lo largo del tiempo y su membresía es independiente de la de cada viaje.

TipoDescripción

15.3 Arquitectura en tiempo real

Principio fundamental: La membresía en Firestore es siempre derivada de PostgreSQL — nunca se gestiona de forma independiente. El frontend lee de Firestore en tiempo real; nunca escribe directamente. Todas las escrituras pasan por la API NestJS.

16. Gamificación

Puntaje · Logros · Chamuco Points · Mapa · Reconocimientos · Retroalimentación

16.1 Traveler Score (Puntaje del Viajero)

Métrica compuesta calculada a partir de: viajes completados, países visitados, km recorridos, logros desbloqueados y retroalimentación recibida. Usado para el ranking global. Nunca editable directamente.

16.2 Logros (Achievements)

Badges automáticos disparados al alcanzar hitos definidos (número de viajes, geografía, distancia, actividad social). Se ganan una vez y nunca se pierden.

16.3 Chamuco Points

  • Moneda blanda dentro de la app. Sin valor monetario real.
  • Se ganan al completar viajes y desbloquear logros.
  • Se gastan en personalizaciones cosméticas del perfil.
  • No transferibles. No expiran.

16.4 Mapa de Descubrimiento

Visualización geográfica personal derivada de los ítems PLACE en viajes completados. Se muestra en el perfil público (si la visibilidad lo permite).

16.5 Reconocimientos (Recognitions)

Badges otorgados por pares. Fuentes posibles:

  • Organizador del viaje al completarlo.
  • Admin del grupo, una vez al año por miembro.
  • Organizador del evento al completarlo.

16.6 Retroalimentación de viaje

Retroalimentación estructurada post-viaje (puntuada + comentario opcional) dirigida a organizadores, más notas opcionales entre pares. Ventana de 7 días después del estado COMPLETED.

16.7 Flujo de completación de viaje

Cuando un viaje pasa a COMPLETED: se actualizan stats, se evalúan logros, se distribuyen puntos, se abre la ventana de retroalimentación y se desbloquea la ventana de reconocimientos para organizadores.

20. Preferencias de Diseño del Usuario

Idioma · Moneda · Tema · Almacenamiento en dos capas

20.1 Opciones de preferencia

PreferenciaValoresDefault
Idioma (AppLanguage)ES Español · EN EnglishES
Moneda de display (AppCurrency)COP Peso Colombiano · USD US DollarCOP
Tema (AppTheme)LIGHT · DARK · SYSTEM (sigue el OS)SYSTEM

La moneda de display solo afecta cómo se muestran los montos — no cómo se almacenan. Los montos siempre se guardan en su moneda original.

20.2 Almacenamiento en dos capas

Capa 1 — Preferencias de invitado (cookie)

Para usuarios no autenticados, las preferencias se almacenan en la cookie chamuco_prefs. Se usa cookie (no localStorage) porque se envía en cada request HTTP, permitiendo que Next.js la lea durante SSR y renderice el tema/idioma/moneda correctos en la primera respuesta sin flash.

CampoValor
Nombrechamuco_prefs
ContenidoJSON: {"lang":"es","currency":"COP","theme":"system"}
Expiración1 año desde la última actualización
FlagsSameSite=Lax, Secure (HTTPS only)

Capa 2 — Preferencias autenticadas (base de datos)

Para usuarios con sesión, las preferencias se guardan en la tabla user_preferences de PostgreSQL. Se cargan una vez tras la autenticación y se cachean en el estado del cliente. Los cambios se persisten vía PATCH /api/v1/users/me/preferences.

20.3 Prioridad de resolución

ContextoOrden de prioridad
Usuario autenticado1. DB → 2. Cookie (solo si DB no tiene registro) → 3. Hint del navegador → 4. Default de la app
Usuario no autenticado1. Cookie → 2. Hint del navegador → 3. Default de la app

20.4 Sincronización al login

SituaciónComportamiento
Cookie tiene valores no-default + DB tiene defaultsSincronizar cookie → DB silenciosamente. El usuario configuró preferencias antes de loguearse.
DB ya tiene valores no-defaultDB gana. La cookie se sobreescribe con valores de DB.
Ambas tienen valores no-default distintosDB gana (refleja una elección deliberada previa del usuario autenticado).

La sincronización es silenciosa y automática — no se muestra ningún prompt al usuario.
